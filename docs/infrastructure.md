# Infrastructure

The shared plumbing under the discobot fleet — the pieces that aren't a bot but
that the bots (and the obsidian-automations supervisor) all depend on. Each is
declarative or self-describing; this page is the map, with a **live uptime
readout** stitched from the metrics we already collect.

## The fleet message bus — Valkey

The loops coordinate over a **Valkey** (BSD Redis) broker — `discobot-valkey` —
the transport half of the [fleet message bus](BUS.md). Pub/sub + retained
last-values carry live telemetry (the supervisor tick → the #ops ferris wheel);
durable streams carry replayable events (reactions, audit). The bus is an
**accelerant, never a dependency**: every publish is fire-and-forget and every
consumer keeps a direct-poll fallback, so an outage degrades rather than cascades.

- **Declared as IaC**, not a `docker run` — Terraform in `dev/infra/valkey/`
  (the `kreuzwerker/docker` provider) owns the container, the `fleet-bus` network,
  and the data volume, because the bus is shared fleet infrastructure rather than
  any one loop's. Apply on the mini: `cd dev/infra/valkey && make apply`.
- **Reachability:** containers join `fleet-bus` and address it by name
  (`redis://discobot-valkey:6379`); host processes (the launchd supervisor,
  Telegraf) use the loopback publish (`127.0.0.1:6379`).
- **Mode:** standalone, RDB-only (`--save 60 1000`, AOF off), `noeviction` — fine
  because retained keys carry TTLs and streams are client-capped, so nothing here
  is a system of record. That's InfluxDB's job.
- **Monitored:** Telegraf's `inputs.redis` scrapes it into the InfluxDB `system`
  bucket (`redis` / `redis_keyspace`, tagged `service=fleet-bus`) — memory,
  clients, ops/sec, keyspace hit-rate, uptime.

## The store — InfluxDB

One shared, multi-producer time-series store (Telegraf + every collector +
tommybot's heartbeat + the supervisor write to it). The bus moves telemetry; the
store keeps it. Dashboards read live off the bus and history off InfluxDB —
neither in the other's hot path. (This is why there's no second database: the
DuckDB-companion idea was parked in favour of one bus + one store.)

## The scheduler — the asyncio supervisor

`obsidian-automations`' supervisor is the fleet's level-1 loop: cron + event
triggers, a memory-admission budget, and a persisted circuit breaker. The
discobots' own `discokit.live` inner loop is a level-2 loop it will supervise.
